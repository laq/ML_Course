Arquitectura de Sistemas

Reactive Manifesto

Los sistemas grandes están compuestos de sistemas más pequeños y dependen de las propiedades reactivas de estos.

  • Responsiveness: is the cornerstone of usability and utility it means that problems may be detected quickly Rapid, consistent, response times, with reliable upper bounds. This simplifies error handling, builds confidence and encourages further interaction.
  • Resilient: Systems stay responsive in face of failure. Replication, containment, isolation and delegation. Recovery is delegated to another (external component)
  • Elastic: System responds under varying workload Increase or decrease allocated resources No bottlenecks
  • Message Driven: Asynchronous message passing Delegate failures as messages. Back-pressure (components must tell when under stress) Non-blocking communication

12 Factor app

I. CodebaseOne codebase in revision control, many deploys
II. DependenciesExplicitly declare and isolate dependencies
III. ConfigStore config in the environment
IV. Backing servicesTreat backing services as attached resources
I: Code
  • All deploys from single code base (many code bases are many apps and each must have 12 factor)
  • Deploy is running instance of app
  • One to one between app and codebase
II: Dependencies
  • Never relies on implicit existence of system-wide packages
  • Dependency declaration and isolation must always be used together (dependency declaration manifest, dependency isolation tool)
  • Dont "shell out" to system tools
III: Config
  • Whatever varies between deploys
  • Litmus Test: Can code base be made public without compromising credentials?
  • Env variables: independent and not grouped
IV: Backing Services
  • Backing service is any service the app consumes over the network.
  • Makes no distinction between local and third party service.
  • Every service is a resource that can be attached or detached at will(no code changes)

12 Factor app

V. Build, release, runStrictly separate build and run stages
VI. ProcessesExecute the app as one or more stateless processes
VII. Port bindingExport services via port binding
VIII. ConcurrencyScale out via the process model
V: Build, release, run
  • Build Stage: transform code to executable(build)
  • Release Stage:Take build and combine with deploy config(release)
  • Run Stage:Runs app in execution environment
  • Every release unique ID
VI: Processes
  • Each app as one or more processes
  • Stateless: don't carry state between transactions
  • Persistence is done through a backing service
  • Stiky sessions violate 12 factor
VII: Port binding
  • Self-contained(no runtime injection into system web server)
  • All processing occurs in user space
  • One app can become backing service for another app.
VIII: Concurrency
  • Scaly horizontally by spawning more app processes
  • App Should never daemonize (Rely on Os system process manager)

12 Factor app

IX. DisposabilityMaximize robustness: fast startup and graceful shutdown
X. Dev/prod parityKeep "environments" as similar as possible
XI. LogsTreat logs as event streams
XII. Admin processesRun admin/management tasks as one-off processes
IX: Disposability
  • Processes can be started or stopped at any time
  • Fast startup
  • Shutdown gracefully with SIGTERM
  • Robust against sudden death (hardware failure)
X: Dev/prod parity
  • Reduce dev and prod gaps:
  • Time gap: write code and deploy it hours or minutes later
  • Personnel gap: Devs who wrote code are involved in deploying it and watching it in prod
  • Tools gap: development and production as similar as possible(also dbs)
XI: Logs
  • App never concerns itself with routing or storage of output stream (STDOUT)
  • Log files configured by execution environment
  • Capture and analyze logs (splunk,Haddop ..) (Find events, graph trends, active alerting)
XII: Admin processes
  • Admin code must ship with application code to avoid sync issues
  • Admin processes should be standalone
  • Run admin scripts with same environment as app (REPL)
